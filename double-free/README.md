# Exhibit: double-free — CWE-415 Double Free

The temporal class the use-after-free exhibit deliberately left out. Revenant (UAF) made a point of
*avoiding* double-free — CLOSE cleared the slot, so a dangling reference could only ever come from a
cache, an alias, or a `realloc`. This exhibit lifts that constraint and studies the lapse directly:
what happens when a pointer reaches `free` a **second** time.

A use-after-free corrupts **user** data — you read or write the dead object, and the damage is whatever
reused that one chunk. A double-free corrupts the **allocator itself** — the second `free` pushes the
same chunk onto the freelist again, poisoning the bookkeeping every other allocation depends on. Same
dangling pointer, different sink; the blast radius jumps from one buffer to the whole heap. And the
classic exploit primitive falls right out of it: a chunk on the freelist twice can be *handed out* twice
(variant 08).

## The grid

**Axis B (rows — temporal shape):** *how* the second free reaches the chunk.
**Axis A (columns — build):** `/Od` debug · `/O2` release · `/O2 /GS` hardened.

| # | Variant | The shape | Sharpest binary tell |
| --- | --- | --- | --- |
| [01](variant-01-df-baseline/) | baseline | `free(p); free(p);` straight line | two `call free` on the same reg/slot, no reassignment between |
| [02](variant-02-guarded-not-nulled/) | guarded-but-not-nulled | `if(p) free(p)` twice — the guard guards nothing | `test/jz` before each `free`, but the pointer is never zeroed |
| [03](variant-03-alias-double-free/) | alias double-free | freed via a local, freed again via a retained global | a `free` of a global/static + a store of a fresh alloc into it |
| [04](variant-04-free-on-error-and-cleanup/) | free on error + shared cleanup | error branch frees, then falls into the `cleanup:` tail that frees again | a `free` dominated by a branch + a `free` at the join |
| [05](variant-05-ownership-transfer/) | ownership transfer | a callee frees its argument; the caller frees it too | `call consume` then `call free` on the same pointer (or both inlined) |
| [06](variant-06-shared-owned-member/) | shared owned member | two structs share one buffer; both destructors free it | `free [base+0]` reached through two different owner objects |
| [07](variant-07-realloc-input-and-output/) | realloc input *and* output | after a successful `realloc`, both old and new pointers are freed | a `free` of the realloc **input** *and* its **output** |
| [08](variant-08-reuse-aliasing-bridge/) | reuse aliasing bridge | free-twice → two `malloc`s alias one chunk → indirect call through the overlap | `free`×2 → `malloc`×2 (same size) → `call qword ptr [reg]` through the reused chunk |

Read top-to-bottom: 01–02 are the undisguised baseline and the defensive-*looking* disguise; 03–07 are
the "looks-correct" shapes where each individual `free` is reasonable cleanup and the defect is only that
a pointer reaches a second free (via an alias, a control-flow join, an ownership contract, a shared
member, or `realloc`'s hidden free); 08 is the exploitation bridge — the double-free becomes a
heap-aliasing primitive that ends in a controlled indirect call.

## What's different from UAF (the lessons that are new here)

1. **The victim is the allocator, not your data.** A UAF read/write corrupts whatever reused the chunk; a
   double-free corrupts the freelist — the structure *every* allocation relies on. So the symptom often
   appears far from the bug (a crash in an unrelated later `malloc`), which makes double-frees nastier to
   trace than their local-looking source suggests.
2. **Unlike UAF, there *is* runtime detection — and it's distinctly Windows.** UAF has no compile-time
   mitigation and frequently no crash. Double-free is partly *caught*: the **debug CRT** (`/MDd`) validates
   the heap and asserts on the second free; the **Windows segment heap / LFH** can detect the freelist
   inconsistency and raise `STATUS_HEAP_CORRUPTION` (`__fastfail`); **Page Heap / Application Verifier**
   (`gflags /p /enable vuln.exe /full`) protects the freed page so the second free faults immediately.
   A plain release build (`/MD`, no Page Heap) is where it stays silent. Knowing *which* layer fires —
   and in what order — is part of the skill.
3. **`/GS` is still irrelevant; `/guard:cf` bites exactly one variant.** No stack array is overrun, so
   `/O2 /GS` is byte-identical to `/O2` (no cookie) across the board. CFG does nothing for 01–07; it bites
   **only variant 08**, where the reused chunk's bytes become an indirect-call target and CFG rejects the
   non-registered destination — and even then it defends only the *call*, not the heap corruption.
4. **Recognition is "two frees, one pointer, nothing between."** Spot a `free`/`HeapFree`, then ask whether
   the same value reaches `free` again with no reassignment in between — directly, through an alias, across
   a branch join, across a function boundary, through a shared struct member, or because `realloc` already
   freed it. At `/Od` the pointer lives in a stack slot; at `/O2` in a non-volatile register across the
   calls.
5. **The fix is ownership and hygiene — the same discipline Revenant used.** Single ownership, null after
   free (`p = NULL` makes a repeat free the harmless `free(NULL)`), don't treat a cache/alias as an
   independent owner, and never free a pointer you handed to `realloc` after it succeeds. Note (variant 06)
   that null-after-free protects only the pointer you cleared — it cannot save an alias.

## Build & run

From **any PowerShell** at the museum root (the script locates the VS toolchain itself):

```
powershell -ExecutionPolicy Bypass -File .\build.ps1 -Exe              # x64 (default)
powershell -ExecutionPolicy Bypass -File .\build.ps1 -Exe -Arch both   # x64 + x86
```

This regenerates each variant's asm (`asm_*.txt` x64; `asm_*_x86.txt` x86) and builds runnable
executables to `double-free\variant-XX\bin\<arch>\<config>\vuln.exe`. Then, e.g.:

```
double-free\variant-01-df-baseline\bin\x64\od\vuln.exe AAAA
double-free\variant-03-alias-double-free\bin\x64\od\vuln.exe AAAA BBBB   (needs two calls)
double-free\variant-04-free-on-error-and-cleanup\bin\x64\od\vuln.exe !AAA   (error path)
```

For a deterministic fault at the second free, run under Page Heap (above). In a plain release build the
double-free may not crash at the point of the bug — that silence is the hazard.

## Caveat: vocabulary vs grammar

Each variant is a small function — it teaches fast, local recognition of a temporal idiom. It can't teach
the *global* lifetime reasoning real double-free hunting demands (who owns the object, who else still
holds a pointer, which call sequence puts the program in the state where the second free fires — variant
03 already needs two calls). See the root [README](../README.md#what-this-museum-teaches--and-what-it-cant).
The module's **capstone — Encore** — will hide these idioms across a realistic Windows application
(shared core DLL + plugins + CLI/IPC/network frontends), so the same recognition has to be paired with
reachability and ownership reasoning across a whole program. It will live in `capstone/` once built;
the withheld answer key stays out of the tree (`capstone/_solution/`, gitignored).
